## Architecture (per the scaffolding plan)

A **src-layout** package. Everything except `tests/` and top-level config lives under `src/`:

- `src/ocr_pipeline/` — capped at 4 files: `ingestion.py`, `preprocessing.py`, `review.py`, `pipeline.py` (orchestrator). No `models.py`, `llm_calls.py`, or `validation.py` — data models that aren't BAML-generated live in the module that produces them, `pipeline.py` calls `baml_client.b` directly (no wrapper module), and business-rule field validation lives in `review.py` alongside the enqueue-for-review logic.
- `src/baml_src/*.baml` — defines the two LLM call shapes as BAML functions: `ExtractOcrBatch` (batched multimodal OCR) and `FillSchema` (text-only structured extraction). BAML's schema-aligned parser generates typed Pydantic output classes — `ocr_pipeline` modules consume those generated types instead of hand-rolled dataclasses/JSON parsing.
- `src/baml_client/` — generated by `baml-cli generate --from src/baml_src`; never hand-edited. The generator's `output_dir` is `../` (relative to `src/baml_src/`) so the client lands at `src/baml_client`, alongside `ocr_pipeline`.
- `pyproject.toml` has a `hatchling` build-system block listing both `src/ocr_pipeline` and `src/baml_client` as wheel packages so `uv sync` installs the project editable and both packages import normally (including from `tests/`).

Pipeline data flow (see the design spec for full rationale): split PDF into pages → detect valid native Unicode text layer per page (catches legacy non-Unicode Bangla fonts that look like text but aren't) → pages without a valid text layer get local image preprocessing (deskew/perspective/contrast) → one batched `ExtractOcrBatch` LLM call transcribes all image-path pages (also returns bounding boxes, captured early for the review UI) → merge native + OCR'd text into one document blob → one `FillSchema` LLM call extracts the fixed schema with per-field confidence → local validation (no LLM, in `review.py`) → anything failing validation or low-confidence goes straight to a human review queue (no automatic re-query/escalation loop by design).

Budget: exactly two LLM call shapes total per document, regardless of page count (~100 pages max) — this is a hard constraint, not an optimization detail.

## Behavioral Guidelines

Behavioral guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

### 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.
